## 🔥 核心原则（永久铁律）

**任何基于数据/消息的研判和决策，必须先核查时效性。**

**时效性 = 决策的前置条件，不是事后检查项。**

---

## ⚠️ 强制流程（违反即报错）

### 1. **任何搜索结果使用前**

必须 100% 确认：

- ✅ **新闻发布日期**（不是事件发生日期）
- ✅ **数据有效期**（财报截止日 / 公告日 / 行情日期）
- ✅ **是否已过期**（监管处罚 > 1 年通常已消化 / 业绩超 2 季度过时）
- ✅ **是否是历史事件被"二次报道"**（自媒体经常炒冷饭）

### 2. **任何"利空/利好"判断前**

必须问自己：

- ❓ 这件事**今天才发生**吗？
- ❓ 还是**已经被市场消化** N 个月了？
- ❓ 是**新公告**还是**旧报道**？

### 3. **任何"立案/处罚/重大事件"前**

必须查：

- 📅 **立案时间**
- 📅 **处罚决定时间**
- 📅 **股价当时反应**（已经跌过 vs 还没反应）

## 🎯 时效性分级

| 等级 | 信息 | 有效期 | 处理 |
|:---:|---|:---:|---|
| 🔴 一手新闻 | 当日发布 | 24h | 必须用 |
| 🟢 近期消息 | 一周内 | 3-7 天 | 可用 |
| 🟡 较旧消息 | 一月内 | 30 天 | 慎用，标注 |
| ⚠️ 历史消息 | 3 个月+ | 已消化 | **必须核对是否 price-in** |
| ❌ 远古消息 | 1 年+ | 极慎用 | 通常**不应作为决策依据** |

## 🛠 技术实现

### 在 stock-realtime-brief skill 中

每次 TinyFish/gsk 搜索结果使用前，必须运行：

```python
def check_freshness(snippet, url, current_date):
    """检查信息时效性"""
    # 1. 提取日期
    date = extract_date(snippet, url)
    
    if not date:
        return {"warning": "无法确定时效性", "use": "需要明确询问用户"}
    
    delta_days = (current_date - date).days
    
    if delta_days > 365:
        return {"warning": "历史事件 > 1 年", "use": "通常 已 price-in"}
    elif delta_days > 90:
        return {"warning": "3 月以上", "use": "慎重"}
    elif delta_days > 30:
        return {"warning": "1 月以上", "use": "可用但标注"}
    else:
        return {"warning": None, "use": "可用"}
```

## 🚨 时间规则补充（2026-05-27 用户强调）

**所有时间表达统一使用北京时间 (CST, UTC+8)**

包括：
- 盘前/盘中/盘后判断
- 距收盘时间计算
- 操作建议时点
- 任何"现在是 X 点" 表述

执行：
- `TZ=Asia/Shanghai date` 是默认指令
- UTC 仅用于内部数据时间戳，不用于显示

## 案例 4：2026-05-29 "卖中天买通富"换仓失误

```
背景:
  Michael 5 月已落袋 ¥103 万
  持有中天 1300 股 浮盈 +¥1.6 万 (+40%)
  
我的建议:
  ❌ "中天是支线，应该卖换 CoWoS 主线"
  
当日结果:
  🚀 中天反弹 +6.79% → 卖飞 ¥3,500
  🔴 通富 -4.20% → 浮亏 -¥0.9 万
  🔴 华工 -6.30% → 同板块同跌
  
错误根源:
  1. 误用 主线识别引擎（仅适用 新建仓）
  2. 忽视 个股浮盈 +40% 已是 winner
  3. 没考虑 同板块集中度
  4. 没把 "不操作" 作为 默认选项

教训:
  ✅ 主线 vs 支线 ≠ 加仓 vs 减仓
  ✅ 已浮盈 winners 不要主动清
  ✅ 同板块集中度 加仓前必查
  ✅ 已落袋 ¥100 万+ = 进入"保护期"
```
